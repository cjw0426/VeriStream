## 1. 问题建模

给定视频 (V=(f_1,ldots,f_T))、问题 (q) 和问题对应的可见时刻 (t_q)，系统只能
访问因果前缀 (V_{leq t_q})。模型单次上下文可以容纳的视觉 token 数为 (B)，而
(Tgg B)。系统需要在预算 (K) 内选择观察工具序列

\[
\pi=(u_1,\ldots,u_K),\qquad u_k\in\{\text{search},\text{inspect},\text{verify},\text{compare},\text{answer}\},
\]

使最终答案 (a=F(q,\pi(V_{leq t_q}),M)) 尽可能准确，同时限制重复视觉编码和上下文增长。
传统 uniform baseline 固定选择帧集合后一次问答；recent4 只保留最新 4 帧；text-memory
将历史帧压缩成文本但不验证其来源。VeriStream 的目标是保存“可回溯证据”，而不是把
未经验证的摘要当作永久事实。

## 2. 方法设计

### 2.1 双角色组织

系统由两个冻结的 Qwen3-VL 角色组成：

1. **感知角色**：问题无关地扫描因果视频前缀，在 anchor 时间点生成粗粒度直接观察；
   它不回答问题，只产生带时间和原始 chunk 指针的观察。
2. **推理角色**：接收问题和检索候选，输出白名单 JSON 工具动作；它决定何时信息充分、
   何时需要局部重观察或核验，并负责最终回答。

第一版使用本地 Transformers 在每个 Accelerate rank 上加载两个模型实例；因此不依赖 vLLM。
后续可将两个角色替换为 OpenAI 兼容 vLLM 服务，但工具权限和 memory 写入仍由本地编排器控制。

### 2.2 分层证据记忆

- **L0 原始证据**：视频路径、chunk 索引、时间区间和原始帧引用；不可被高层记忆淘汰。
- **L1 观察记忆**：观察文本、实体、动作、置信度和状态 `candidate/verified/quarantined`。
- **L2 事件记忆**：按实体、动作和相邻时间将多个 L1 观察聚合为状态或事件摘要。
- **L3 问题工作集**：当前问题检索到的少量 L1/L2 ID，问题结束后清空。

写入时，同一视频、重叠 chunk 且文本 Jaccard 相似度超过阈值的观察合并；冲突观察保留
为不同版本，不直接覆盖。容量不足时优先淘汰未验证、低访问、低置信度且较早的观察。
跨视频检索在 `video_id` 命名空间层面禁止。

### 2.3 检索与工具

`search_memory` 使用词元重合、实体/动作匹配、时间过滤、置信度和 verified 状态奖励排序。
推理角色可以调用：

```text
search_memory(query, top_k)
inspect_segment(start_time, end_time, target)
verify_evidence(memory_ids, target)
compare_segments(memory_ids, target)
answer
```

`inspect_segment` 将局部原始帧送回感知角色并写入新观察；`verify_evidence` 回取 L0 原始帧，
要求模型输出 `SUPPORTED/UNSUPPORTED/CONFLICT`，只有支持的观察才能进入最终上下文；
`compare_segments` 用少量局部帧判断状态变化。模型只输出 JSON，Python 编排器校验白名单
和参数后才执行工具。

当前实现的 `search_memory` 是本地确定性检索，不调用另一个语言模型或向量数据库：先对问题、
观察文本、实体和动作做中英文词元切分，计算交集数量，再加入置信度、访问次数和已核验状态
奖励，按分数排序返回 Top-K。它的优点是可解释、无需额外模型；局限是同义词和跨语言表达
可能无法匹配，后续可增加冻结 embedding 作为召回层，但不能把模型自由生成结果直接当作证据。

对 observation 的实际排序分数为：

```text
lexical_overlap = |query_tokens ∩ (observation_tokens ∪ entity_tokens ∪ action_tokens)|
score = lexical_overlap + 0.25 * confidence + 0.02 * access_count
        + 0.6 * 1[status == VERIFIED]
```

`confidence` 位于 `[0, 1]`；`access_count` 每次被选入检索结果后加一；被隔离、淘汰或过期的
记忆不参与检索。事件级记忆使用相同的词元重合思想，但置信度权重为 `0.2`、已核验奖励为
`0.5`，不使用访问次数。分数只决定候选顺序，不代表事实概率。

### 2.3.2 核验门控与事实准入

候选检索分数只用于排序，不能决定事实是否可信。系统采用独立的核验门控：

1. **触发**：推理控制器看到候选记忆未验证时，应输出 `verify_evidence`；如果控制器提前输出
   `answer`，但当前工作集没有任何 `VERIFIED` 记忆，编排器会自动核验工作集中的最高相关候选。
2. **回取原始证据**：编排器根据 `memory_id` 中保存的 `chunk_indices` 回取 L0 原始帧；找不到
   原始帧时直接将该记忆标为 `QUARANTINED`。
3. **视觉判定**：感知模型必须在第一行输出 `SUPPORTED`、`UNSUPPORTED` 或 `CONFLICT`，第二行
   给出直接可见事实。解析时只有第一行以 `SUPPORTED` 开头才算通过；空输出、格式错误、
   `UNSUPPORTED` 和 `CONFLICT` 全部视为不通过。
4. **状态更新**：通过的记忆标为 `VERIFIED` 并记录核验时间；不通过的记忆标为
   `QUARANTINED`，后续检索会排除它，但其 L0 时间指针仍保留，便于审计和重新观察。
5. **最终准入**：最终回答 prompt 只接收状态为 `VERIFIED` 的记忆，不接收 candidate、quarantined
   或 evicted 记忆；没有已验证证据时必须向用户说明证据不足。

当前实现没有把 `confidence` 设置为硬阈值，也没有用候选分数直接放行；即使置信度很高，仍需
   原始帧核验。一次 `verify_evidence` 最多处理 `max_working_memories` 条记忆，工具循环次数
   受 `max_actions` 限制。该设计将“相关性召回”和“视觉事实确认”分离，避免文本摘要污染最终答案。

### 2.3.1 两类文本记忆的区别

| 机制 | 记忆生成提示重点 | 适合回答的问题 | 主要风险 |
| --- | --- | --- | --- |
| 动作事实文本记忆 | 历史帧直接可见的对象、动作、位置和对象状态 | “谁做了什么”“物体出现或移动在哪里” | 时间连续性较弱，可能把历史动作误当成当前状态 |
| 状态感知文本记忆 | 过去状态、场景连续性、已完成进度及状态变化 | “后来变成什么状态”“动作前后有什么变化” | 描述更抽象，可能丢失细粒度动作 |

两者都是一次性历史片段文本压缩，区别在感知提示的字段和关注目标，不涉及训练。双角色
VeriStream 使用自己的结构化观察索引，不把历史目录后缀当作方法名。

### 2.4 推理算法

推理模型不是被动接收固定帧，而是在每一步根据问题和候选证据自主选择工具，最多循环
`max_actions` 次；工具结果会回填下一轮控制器上下文。控制器实际使用的核心提示是：

```text
You are the reasoning controller for long-video question answering. Candidate summaries are not final facts;
verify any unverified summary before relying on it.
Return exactly one JSON object with no Markdown. Allowed actions: search_memory, inspect_segment,
verify_evidence, compare_segments, answer.
Format: {"action":"...","memory_ids":["..."],"query":"...",
"start_time":null,"end_time":null,"target":"...","answer":"..."}
```

`inspect_segment` 输入时间区间和关注目标，感知模型重新读取该区间最多 8 帧，提示为“请重新
观察给定局部视频帧，只陈述直接可见事实”，它负责**获取新证据**。因此模型可以多次重新观看
不同片段，但次数受 `max_actions` 预算限制。

`verify_evidence` 输入已有 `memory_id`，感知模型读取其对应原始帧，提示要求第一行输出
`SUPPORTED`、`UNSUPPORTED` 或 `CONFLICT`，它负责**核验已有证据**而不是生成新记忆。只有
`SUPPORTED` 才将记忆标为 `VERIFIED`，否则进入隔离状态。若控制器提前回答且没有已核验证据，
编排器会自动核验最高相关候选；最终答案 prompt 只接收已核验证据，并要求证据不足时明确说明。

```text
输入：因果视频 chunk、问题 q、预算 K
1. 感知角色按 coarse_stride 建立 L0/L1/L2 视频索引
2. working_set <- search_memory(q)
3. for step = 1 ... K:
       action <- 推理角色(JSON)(q, working_set, 最近工具结果)
       若 search：检索并更新 working_set
       若 inspect：重观察时间段，写回 L1/L2，更新 working_set
       若 verify：读取原始帧，SUPPORTED 则升级，否则隔离
       若 compare：比较候选证据的局部帧
       若 answer：停止循环
4. 若没有 verified 证据，安全核验最高相关候选
5. 推理角色基于 q + verified 证据生成最终答案
```

## 3. 实验设置

### 3.0 评测 Harness 的执行边界

`main_experiments/eval_qwen3vl_ovo_dual_role.py` 是评测 harness，负责数据、模型、并行和计分；
`lib/veristream_dual_role.py` 是 agent 核心，负责记忆、工具和证据门控。一次样本的完整顺序为：

```text
OVO 标注
  -> harness 按 realtime 边界因果解码视频 chunk
  -> 感知角色扫描 chunk，写入/加载该视频的持久 memory JSON
  -> agent 用问题检索候选记忆
  -> 推理角色循环选择 search / inspect / verify / compare / answer
  -> 感知角色按需重新读取局部原始帧
  -> agent 只将 VERIFIED 观察交给最终推理 prompt
  -> harness 将答案写入 rank JSONL
  -> rank0 合并所有 rank 并调用 OVO scorer
```

感知角色和推理角色在代码中都是冻结的 `RecentWindowQAModel` 实例，但职责不同：感知角色的
`generate_from_frames` 接收图像并输出观察或核验标签；推理角色的 `generate_from_text` 只接收
问题、记忆文本和工具结果，输出 JSON 工具动作或最终答案。当前默认每个 Accelerate rank 在一张
GPU 上各加载一个模型实例；`--share-model` 可让两角色共享实例以降低显存，但不改变编排逻辑。

Harness 使用 round-robin 将 Backward/Realtime 样本分给各 rank，每处理一个样本立即追加到
`rank_{rank}.jsonl`，并把视频 memory 保存到 `memory/{video_id}.json`。rank0 等待所有
`.done` 标记后合并结果；因此中断后可以复用已经完成的 rank 行和 memory，而不会重复建立索引。

### 3.1 数据与指标

主表使用 OVO-Bench 共 1,640 个问题：Backward 631、Realtime 837、Forward 172。
当前双角色入口只运行 Backward 和 Realtime；历史基线含完整三 split。每个 split 先按
任务类型计算准确率，再对任务类型平均；Total 是三个 split 平均值的平均，不是按问题数加权。

模型为 Qwen3-VL-2B/8B-Instruct，冻结参数，不进行训练。硬件为 8×RTX A6000，随机种子
为 42，视频解码默认 `chunk_duration=1.0, fps=1.0`；双角色快速实验使用
`chunk_duration=2.0, fps=0.5, coarse_stride=12`，该配置应在表中单独标注。

### 3.2 实验命名说明

历史结果目录中的 `v3`、`v4` 只是实现迭代编号，不是新的模型或训练阶段。本文统一使用机制名：

- **动作事实文本记忆**：代码入口 `recent_memory_uniform` / `recent_memory_clip_topk`。
  历史目录中的 `memory_*_v3` 属于这一类；先对历史帧生成“动作、位置、对象状态”等事实描述，
  再压缩成少量文本条目。
- **状态感知文本记忆**：代码入口 `recent_state_memory_uniform_v4` /
  `recent_state_memory_clip_topk_v4`。历史目录中的 `state_memory_*_v4` 属于这一类；提示词
  明确要求记录过去的对象状态、场景连续性和已完成进度，避免把过去动作误判为当前动作。

两类 memory 的共同生成流程是：先保留最近 `recent_frames_only=4` 帧作为当前视觉窗口，
再从更早的历史帧中选择 `supplemental_frames=16` 帧。`uniform16` 在历史前缀上均匀取样；
`CLIP-topk16` 使用冻结 CLIP 与问题文本计算相似度，选取最相关的 16 帧。历史帧按
`memory_group_size=4` 分组，每组由 Qwen3-VL 生成一句英文 caption，再由文本模型将全部
caption 压缩为最多 `memory_num_items=3` 条短 bullet；最终问答仍使用最近 4 帧和这些文本条目。

两类记忆只改变历史 caption/summary 的感知目标：

| 记忆类型 | 感知 prompt 关注内容 | 设计动机 |
| --- | --- | --- |
| Recent4 + action-fact text memory | 直接可见的动作、对象位置、对象状态和时间事件 | 保留“发生了什么”的事件事实，服务动作追踪和位置回溯 |
| Recent4 + state-aware text memory | 过去对象状态、场景连续性、角色身份和已完成进度 | 防止把过去动作误认为当前动作，服务状态变化和长期进度问题 |

两类感知 prompt 的核心文字如下：

```text
Action-fact caption prompt:
Describe only directly visible historical facts from these frames in one short sentence.
Focus on concrete actions, object locations, object states, and temporal events.
Do not answer any question, guess hidden facts, or state a final conclusion.

State-aware caption prompt:
Describe only past visual state from these historical frames in one short sentence.
Focus on object identity, object locations, scene continuity, actor identity, and completed progress.
Avoid framing past observations as evidence that a queried action is happening now.
Do not answer any question or state a final conclusion.
```

例如同一段历史帧显示“一个人把红杯子放到桌上，随后离开”：

- Action-fact 可能生成：`A person places a red cup on the table and walks away.`
  它强调发生的动作、对象和位置，适合回答“谁做了什么”或“杯子被放在哪里”。
- State-aware 可能生成：`The red cup is left on the table after the person completes the placement.`
  它强调动作完成后的状态和进度，适合回答“之后杯子处于什么状态”。

如果当前问题是“此刻这个人是否正在放杯子”，Action-fact 摘要更容易让模型回忆到“放置”
这个历史动作；State-aware prompt 则要求保留“动作已经完成、杯子留在桌上”，降低把过去动作
误判为当前动作的风险。两者的差异来自 prompt 约束和压缩目标，不来自额外训练。

提出这两类记忆是因为均匀增加原始历史帧会带来冗余和上下文压力，而一次性摘要又可能混淆
“历史动作”和“当前状态”。动作事实记忆优先保留事件细节，状态感知记忆优先保留时间连续性；
它们是 training-free 的 prompt-level 对照，用于分析 memory 粒度和语义目标，而不是两个训练模型。

### 3.6 双角色失败分析与下一方向

本次失败不是单一 prompt 问题，而是解耦架构中不同信息通道失衡：

1. **历史与当前信息不对称**：L0-L2 覆盖长视频，但最终推理只接收文本 memory，没有接收
   realtime 边界附近的原始帧。HLD 从 52.15 提升到 74.73，说明历史证据对长期位置/事件
   有帮助；STU、OJR、ATR、FPD、ACR、OCR 全部下降，说明当前状态、细粒度动作、空间关系
   和画面文字不能稳定地由粗粒度文本替代。
2. **感知压缩损失不可逆**：一个 observation 将多帧视觉内容压缩成一句自然语言，容易丢失
   小目标、左右关系、计数、OCR 字符和动作顺序，后续推理无法从摘要中恢复这些信息。
3. **核验门控过于保守**：约 18.7% 样本最终输出 “Evidence is insufficient.”。`VERIFIED`
   约束提高了事实安全性，却在候选不完整时阻止模型利用仍然有用的当前视觉线索。
4. **工具回跳无法补回缺失上下文**：`inspect_segment` 只能观察控制器指定的时间段；如果问题
   需要 realtime 边界附近的连续状态，文本控制器可能没有足够线索选对区间。继续增加
   `max_actions` 会增加成本，但不保证提升准确率。
5. **实验实现混杂因素**：本轮使用 4 个 rank 而不是 8 个，并且每个问题独立建立视频索引，
   主要放大运行时间；另有 2 个 ffmpeg 资源错误，需单独统计，不能把它们当作模型判断错误。

已实现的下一版本不是放弃 VeriStream，而是恢复两个互补的信息通道：

```text
全视频感知 -> L0-L2 持久 memory -> 检索/核验 -> VERIFIED 历史证据
                                             + 最近 4 帧原始视觉输入
                                             ↓
                                      多模态最终推理
```

采用“已验证 memory + 最近 4 帧”的理由是：

- **保留长期能力**：HLD 的提升说明 memory 能补充最近窗口看不到的历史事件和位置变化。
- **恢复当前视觉能力**：最近 4 帧与 strong recent4 baseline 对齐，提供当前状态、动作结果、
  空间关系和 OCR 线索，直接针对 Real-time 退化。
- **保留事实安全性**：历史信息仍需通过 `verify_evidence` 才能进入 VERIFIED memory；最近帧
  是原始视觉证据，不必先压缩成文本。
- **保持创新边界**：感知/推理解耦、持久 memory、工具调用和核验门控都保留，只将最终推理
  从 text-only 改为“文本证据 + 少量当前视觉”的多模态上下文。

需要通过以下消融验证，而不能直接声称有效：

| 变体 | 历史 VERIFIED memory | 最近原始帧 | 目的 |
| --- | --- | --- | --- |
| Text-only DualRole | 有 | 无 | 当前失败版本 |
| Hybrid-4 | 有 | 最近 4 帧 | 当前默认实现，验证当前视觉是否恢复 Real-time |
| Hybrid-8 | 有 | 最近 8 帧 | 测试当前视觉预算敏感性 |
| Recent4 only | 无 | 最近 4 帧 | 与 baseline 对齐 |
| Hybrid-4 without verification | 候选 memory | 最近 4 帧 | 测试核验门控是否过严 |

核心成功标准是：Hybrid-4 的 Real-time 显著高于 Text-only DualRole，同时 HLD 不明显低于
当前 74.73；若收益只来自最近 4 帧而 memory 没有贡献，则应重新设计 memory 粒度和检索，
而不是继续增加工具回跳。

## 4. 消融与观察到的现象

### 4.1 已有对照所体现的机制消融

1. **原始帧历史 vs 文本记忆**：8B recent4+uniform28 的 Total 为 57.27，低于 recent4
   的 58.26；text-memory uniform16 提升到 60.26。说明压缩历史本身比简单增加视觉帧更有效。
2. **检索选择**：8B text-memory uniform16 为 60.26，CLIP-topk16 为 59.95；当前数据上
   均匀历史压缩略好于 CLIP top-k，说明视觉相似度不等于问题相关性。
3. **历史信息的实时性代价**：8B text-memory uniform16 将 Backward 从 53.92 提升到
   62.09，但 Real-time 从 81.47 降到 79.65；历史信息收益和当前状态污染之间存在权衡。
4. **模型规模**：2B recent4 的 Total 为 56.82，8B recent4 为 58.26；2B text-memory
   的 Backward 可以达到 57.30，但 Forward/Real-time 仍明显受限。

### 4.2 双角色需要报告的消融

完整双角色结果生成后，应至少比较：

- Hybrid-4（VERIFIED memory + 最近 4 帧）与旧版 text-only 双角色；
- 粗粒度索引 + 检索 + 核验（完整方法）；
- 去掉 `verify_evidence`，候选摘要直接进入答案；
- 去掉 `inspect_segment`，只使用预计算索引；
- 仅保留 L1 观察、不构建 L2 事件；
- 共享模型实例 vs 两个模型实例；
- `max_actions=2/4/6` 的准确率-延迟曲线。

这些消融需要在 `eval_qwen3vl_ovo_dual_role.py` 增加显式开关后再运行；当前代码已固定
完整工具路径，尚未把未运行的消融写成结果。

## 5. 问题、失败与修复

- **旧版 NCCL barrier 超时**：某 rank 完成 Forward 后等待超过 600 秒，导致所有结果丢失。
  双角色脚本改为 `.done` 文件同步，rank 完成后立即退出，rank0 轮询文件再合并。
- **Accelerate 小样本复制**：旧版本 `split_between_processes` 在样本少于 GPU 数时复制末尾
  样本。双角色入口改为显式 `rows[rank::world_size]` round-robin，并在 smoke 中验证无重复 ID。
- **双模型 OOM**：旧版全量任务占用约 17GB/卡时，再加载第二模型会只剩几十 MB。正式双角色
  必须等待旧任务退出；A6000 空闲时双模型约占 35GB/卡。
- **NCCL 机器配置**：该机器必须设置 `NCCL_P2P_DISABLE=1` 和 `NCCL_IB_DISABLE=1`。
- **视频解码异常**：部分 OVO mp4 触发 decord EOF 或 scaling graph 警告；评测器应保留单样本
  错误并计 0，不能让全轮退出。错误数量、视频 ID 和回退 backend 必须写入结果。
- **长视频计算量**：感知、控制器、核验和最终回答均可能调用模型。报告中需要额外统计
  `perception_calls`、`reasoning_calls`、工具步数、平均延迟和 memory 观察数。

## 6. 当前结论与待完成项

已有 training-free baseline 支持以下暂时结论：文本记忆比简单增加历史原始帧更有效，但未经核验的历史
信息会损害 Real-time；因此“带来源指针、按需回溯和验证门控”的双角色设计具有明确动机。
当前不能声称 VeriStream 已经提升准确率，因为 2026-07-15 双角色全量评测尚未结束。

待完成：双角色完整 Backward/Realtime 分数、验证/隔离比例、平均工具步数、延迟统计、
去核验和去局部观察消融，以及将最终结果写回 `实验报告.md`。
